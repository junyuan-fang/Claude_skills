# Learning with Code + Paper

把"一个开源项目 + 它的论文"或"一个领域/概念 + 教材与示例代码"炼成一套结构化课程,产出物是 md 源文件 + 一个可离线打开的本地文档网站。这是从 cuRobo IK 学习项目里提炼出的方法论(该项目在 `/data/curobo`,可作参考实现直接去看)。

## 两种模式

先判断用户要哪一种(不确定就问):

- **模式 A:repo + 论文**(原始场景)。目标是吃透一个具体开源项目及其论文,直到能读懂/改代码/贡献 PR。产出以"阶段"(stage,对应代码模块)为主线。
- **模式 B:领域/概念学习**(新增)。用户没有指定具体 repo,而是想系统学会一个领域或方向(如"贝叶斯统计"、"强化学习"、"因果推断")。目标不是贡献某个仓库,而是**吃透这个领域的知识体系,并具备该领域的学术思维**——知道核心问题是什么、有哪些学派/方法论之争、经典结果怎么证明、前沿在哪、遇到新论文能自己判断好坏。产出以"主题/概念"为主线,配教材而非单一论文,配多个小的示例代码/演示(不必来自同一个 repo,可以是教材配套代码、经典库的最小复现、自己写的教学脚本)。

两种模式共享同一套质量红线、页边注/术语系统、KaTeX 离线网站生成管线(`assets/build.js`)。差异主要在"开工前确认什么"和"产出结构"上,见下文对应小节。

## 何时用 / 先问清楚什么

触发词:"帮我学这个 repo"、"整理学习路线"、"我要学会 X 然后贡献"、"我想系统学习 XX 领域/方向"、"做成课程/网站"。

先确认走模式 A 还是模式 B(见上一节),再按对应清单确认:

**模式 A(repo + 论文)开工前确认**:
1. **目标仓库/论文**:repo URL(或本地路径)+ 论文(可能不止一篇——用户说"vX 论文"时不要假设是哪篇,先搜清楚版本关系,如实呈现,不要编)
2. **读者水平**:纯小白 / 有基础。决定是否需要"零基础白话导读"这类零数学入口页
3. **深度目标**:只是理解 vs 要贡献代码(决定是否要"贡献之路"阶段、代码调用链深度)
4. **产出形式**:是否要网站(默认要,离线优先),还是只要 md

**模式 B(领域/概念学习)开工前确认**:
1. **具体领域/方向范围**:用户举的例子(如"贝叶斯方向")往往只是举例,不要默认就是唯一目标——用 AskUserQuestion 确认具体边界(如"贝叶斯统计"是指贝叶斯推断基础 + MCMC,还是也要贝叶斯深度学习?),避免范围失控或猜错
2. **权威教材选型**:该领域公认的 1-3 本经典教材/讲义是什么(如贝叶斯统计常见 Gelman *Bayesian Data Analysis*、McElreath *Statistical Rethinking*、Murphy *Probabilistic Machine Learning*)——不确定就先派 teammate 搜清楚该领域的教材共识,如实呈现候选并说明各自侧重,不要凭印象编一本
3. **读者水平**:纯小白 / 有基础(如已经懂微积分线代概率论,还是要从头建立直觉)。决定"零基础白话导读"要铺多厚
4. **深度目标**:只是"看懂论文/懂概念" vs "要能自己推导证明、复现经典结果、对新论文有批判性判断"(后者需要更多"📐 推导"、"学术视角 vs 工程视角"、"流派对比"内容)
5. **示例代码来源**:是否有指定的参考代码库(如 PyMC、Stan、numpyro),还是由你选/写最小可跑示例来配合每个概念
6. **产出形式**:同模式 A,默认要离线网站

## 全局质量红线(每一条都来自真实踩坑,别再踩)

1. **公式用主流学术记号,不用自造伪记号**。优化类问题遵循 Nocedal & Wright《Numerical Optimization》的记号(θ_k, p_k, α_k, ∇C, Hessian, s_k/y_k);领域记号遵循该领域最标准的教材/论文(机器人学如 *Modern Robotics*;统计/贝叶斯如 Gelman *BDA* 或 Murphy *PML*;其它领域现找现查,不要凭印象)。开一个「符号约定表」放在最先读的理论文件顶部,其它文件锚点引用,不要重复定义。
2. **每个方法配一段"简单推导"**,不是甩公式。格式:「📐 推导」引用块,一句目标 + 3–8 行不跳步 + 一句结论。参考已有的牛顿法(1 维抛物线拟合直觉 → n 维)、高斯-牛顿(残差线性化 → normal equations)写法。
3. **术语缩写必须展开到源头,并给权威链接**。不是随手写个 Wikipedia 链接就完事——**每条链接都要实际验证能打开**(WebFetch 或 curl 测一下状态码/内容),死链、404、被墙的域名(如 modal.com 系)一律不用。倾向 Wikipedia / arXiv 原文 / 官方文档 / DOI 永久链接。缩写要给"全称是谁提出的"(如 L-BFGS 四位姓氏、LM = Levenberg 1944 + Marquardt 1963),不要只写展开不给出处。
4. **论文与代码若有出入,如实标注,不要为了叙事整洁而抹平**。例如论文报告的超参数和当前代码配置不同,两个都写出来并说明"属正常重新标定"或"存疑",这正是训练读者的工程判断力。模式 B 同理:教材之间记号/假设不一致、某方法在实践中的近似跟理论表述有差距,都如实标注。
5. **零基础与深度不冲突,分层放**:白话类比版(如"IK=蒙眼多人下山")放最前当可选入口,不替代严谨版;严谨版才是主体。用户反馈"看不懂"时,翻译成白话+比喻,同时补一篇独立的"零基础导读"文件,不要把主文件也降级。
6. **网站必须完全离线**:字体、KaTeX、JS 一律本地 vendor,不用 CDN。`assets/vendor/katex/` 已经打包好可以直接复用。
7. **每次改完 UI/生成脚本,用无头 Chrome 做一轮硬验收**:控制台零报错、多宽度(375/768/1024/1440/1920)无横向滚动、明暗双主题对比度、内部链接全部可解析、外链全部可达。不要只凭"应该没问题"交付。
8. **(模式 B 专属)训练学术思维,不只是灌知识点**:每个理论文件除了"是什么/怎么推",还要点出——这个方法解决了什么问题、有没有竞争方法/学派分歧(如贝叶斯 vs 频率派在同一问题上怎么答)、经典结果的原始出处是谁、当前领域公认的开放问题或局限是什么。目的是让读者读完不仅会用公式,还知道"这个领域的人是怎么思考和争论的",能对新论文形成自己的判断,而不是死记结论。

## 产出结构(照抄这个目录约定)

模式 A(repo + 论文):

```
<project-root>/
├── <project>-learning/                    # 学习资料根目录(学 curobo 时叫 curobo-learning)
│   ├── <总索引>.md                        # 阶段表 + 理论表 + 三条原则 + 三个坑
│   ├── 资源链接.md                        # 视频/教材外链,分类,标★优先级
│   ├── 术语表.md                          # 全部术语人类可读版(从 glossary.js 同源生成)
│   ├── papers/                            # 下载好的论文 PDF
│   ├── stages/                            # 00, 01, 02… 分阶段学习路线
│   │   └── NN-<主题>.md                   # 见 templates/stage-template.md
│   ├── theory/
│   │   ├── 00-零基础白话导读.md            # 可选,零数学
│   │   └── NN-<概念>.md                   # 见 templates/theory-template.md
│   └── site/
│       └── vC/                            # 网站产物(名字随意,但保持一个"当前版本"目录)
│           ├── index.html, stage-N.html, theory-N.html, glossary.html, resources.html
│           ├── style.css, app.js, search-index.js, glossary.js
│           └── vendor/katex/               # 直接复制 assets/vendor/katex/
└── <repo>/                                # git clone 的目标仓库
```

模式 B(领域/概念学习,无固定 repo)——目录约定基本一致,把"阶段=代码模块"换成"阶段=知识主题",把 `papers/` 换成 `textbooks/`(教材/讲义 PDF 或抄录的关键章节+页码指针),`stages/` 里每个文件对照的是"教材章节 ↔ 示例代码"而不是"论文页码 ↔ 仓库路径`:

```
<project-root>/
└── <domain>-learning/                     # 如 bayesian-learning
    ├── <总索引>.md                        # 主题地图(建议依赖关系图/学习顺序) + 理论表 + 三条原则 + 三个坑
    ├── 资源链接.md                        # 教材/课程/视频外链,分类,标★优先级
    ├── 术语表.md                          # 全部术语人类可读版(从 glossary.js 同源生成)
    ├── textbooks/                         # 教材/讲义 PDF 或关键章节摘录(注明版次、章节号)
    ├── examples/                          # 示例代码:教材配套代码、经典库最小复现、自写教学脚本
    │   └── NN-<概念>/                     # 每个概念一个可独立跑的最小示例,而非单一 repo 的调用链
    ├── stages/                            # 00, 01, 02… 按知识依赖顺序排列的主题路线
    │   └── NN-<主题>.md                   # 结构同 templates/stage-template.md,"代码侧"换成"示例代码 + 教材对照表"
    ├── theory/
    │   ├── 00-零基础白话导读.md            # 可选,零数学
    │   └── NN-<概念>.md                   # 见 templates/theory-template.md,额外要求见质量红线第 8 条(学术思维)
    └── site/
        └── vC/                            # 网站产物,结构与模式 A 完全一致,直接复用 build.js
```

## 执行流程(多 teammate 编排)

小任务自己做;涉及"调研 + 写作 + 前端实现 + 审查"这种能并行、能独立验证的组合,用 Agent 工具派出命名 teammate 并行,SendMessage 做阶段间握手(例如"md 定稿"信号触发"网站重建")。参考角色分工(来自 cuRobo 项目实录,模式 B 的角色变体见括号):

| 角色 | 职责 | 何时并行 |
|---|---|---|
| `paper-scout`(模式 B 叫 `textbook-scout`) | 模式 A:确认论文版本、下载 PDF、产出算法向摘要(带页码/章节指针)。模式 B:调研该领域公认的经典教材/讲义有哪些、如实呈现候选(不确定哪本"最经典"就都列出并说明各自定位),下载/摘录关键章节,产出"主题 ↔ 教材章节"指针表 | 与 code-mapper 同时 |
| `code-mapper`(模式 B 叫 `concept-example-coder`) | 模式 A:梳理代码调用链(入口→核心算法→底层实现),给出文件路径+行号。模式 B:为每个核心概念挑选或编写一个可独立跑的最小示例代码(可以来自教材配套代码、经典库如 PyMC/Stan/numpyro 的最小复现、或自己写的教学脚本),不追求单一仓库的完整调用链 | 与 paper-scout 同时 |
| `math-editor` | 把 md 里的公式改成主流记号,加符号约定表、学术/工程双视角、📐 推导块;模式 B 额外要求补"流派对比/开放问题"(见质量红线第 8 条) | 内容侧收尾 |
| `glossary-researcher` | 核实术语全称、出处、权威链接(**必须实测链接可达**) | 可与 math-editor 并行 |
| `frontend-builder` | 实现/维护 build.js,生成静态站点(页边注、KaTeX、搜索、进度) | 等内容侧产出 md 后重建 |
| `reviewer` | 无头 Chrome 硬验收(见质量红线第 7 条),发现问题直接修复 | 每轮重建后 |

**编排要点**:
- 不确定"论文 vX 具体指哪篇"(模式 A)或"该领域最权威的教材是哪几本"(模式 B)这类事实问题,先派 teammate 用 WebSearch/WebFetch 查清楚,如实汇报候选项,不要自己猜一个塞进去。
- 前端团队和内容团队可以并行:前端先用假数据把 UI 跑通,内容定稿后发"重建信号"再跑真数据全量重建。
- 大改动(比如用户否掉了前一版设计,或模式 B 下用户否掉了主题范围)先用 AskUserQuestion 收敛方向,不要闷头再做一版容易再被否的。
- 每个 teammate 收工后简报交付清单+自验证据(不是"我做完了",而是"跑了 node --check / headless Chrome / curl 状态码,结果是……")。

## 复用资产

- `assets/build.js` —— cuRobo 项目的完整生成脚本(Node,无依赖),把 md 编译成 VitePress 风格站点:markdown→HTML、KaTeX 公式占位符保护、术语首次出现标注、search-index 生成、sidebar/pager/TOC。**改造成新项目时**只需改顶部的 `ROOT`/`OUT`/`PAGES` 数组(文件路径、标题、分组),转换逻辑基本不用动。
- `assets/vendor/katex/` —— KaTeX 0.16 完整离线包(css/js/auto-render/字体),直接复制到新站点的 `vendor/katex/`,不要重新下载。
- `templates/stage-template.md`、`templates/theory-template.md` —— 空白模板,照着填。
- `templates/glossary-entry.example.js` —— 术语条目的字段格式范例。
