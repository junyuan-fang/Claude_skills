# 用户画像

## 偏好风格
- 偏好简洁汇总，不要中间态/工具调用步骤的状态刷屏，多步任务只在末尾一句总结
- 希望推送内容有深度展开（背景/技术/数据/业内对比），而不是只列关键词
- 喜欢用具体数字（金额、产能、参数量）和权威源链接支撑要点
- 推送内容里公司/产品业务背景必须带，但已熟的技术概念不要再解释
- 视频/竖屏内容偏好 YouTube Shorts 形态（1080×1920 / 9:16 / 60-90s）
- 名词笔误/缩写场景偏好"列候选 + 让用户确认"，而不是猜一个就硬答
- 概念扩展期望结构化展示（表格列名字/类型/用途），并补主要作者、所属机构、GitHub repo
- 公众号/长文解读期望结构化摘要：核心事件 + 技术三件套表格 + 关键数字 + 作者金句 + 一句话判断
- 解释 RL/critic 等基础概念时偏好"最朴素定义 → 对应关系类比表 → 文章语境下的变体清单"三段式
- 多步流水线（如 daily-papers 三步走）结束时只要一句话总结（产物路径 + 数量 + 关键 flag 状态）
- daily-papers 一句话总结里期望额外补必读/值得看/可跳过的分层数量与 MOC 篇数

## 沟通习惯
- 中文沟通，语气随意口语化（"你看着办"、"反正就类似这些"）
- 倾向迭代式反馈：先看效果再提改进，常要求"再跑一次看看"
- 会顺手抓事实错误（比如论文年份不对），期望立即修正并加防护规则
- 会主动追问实现细节（"实现方式发我看看"），期望结构化展示模块/流水线分层
- 偶尔只发"你好"等闲聊性短消息，期待简短回应 + 当前任务状态摘要
- 提问常用极简打字（拼错或缩写，如"nv diff 的"），期望对方主动补全并展开
- 跟进追问常常是上一轮的精确指代（例如先问"nvidiafffinc"，再发"nv diff 的"补正），需要把跨轮上下文串联
- 会丢公众号链接（mp.weixin.qq.com）让对方读全文并产出结构化解读
- 同一会话里读完长文后常会追问其中的某个术语（如"critic 是什么"），期望基于上文语境精准解释，不要离题
- cron 调度调整诉求短促直接（"改成工作日才启动"），期望对方一次到位改完并给确认表
- 长流水线任务（如 daily-papers）结尾期望一句话报路径+数量+关键 flag 状态

## 注意事项
- 多步/后台任务执行中不要逐步播报进度，结束时给一条总结即可
- 新闻类 cron 红线：单图、不重试、sleep 30 之间隔、不发任何收尾 reply 状态消息
- 涉及"飞书"用语时确认实际通道，避免误导
- 解释概念时严格 gating：能搜到 Wikipedia 条目的不解释，只对新论文内部命名/小众 benchmark 1 句带过
- 公司业务背景（谁创立、估值、做什么）不属"已熟"，仍需带
- 引用 arXiv 论文为"今日新发"必须 14 天内：HuggingFace daily 优先，arXiv ID YYMM 粗筛 + abstract 页 `[Submitted on]` 精校；搜不到就放弃论文项，不包装老论文
- 用户笔误/含糊提问时，先列 2-4 个最可能候选并简表对比，再请用户确认，避免一上来猜单个目标
- 用户极简追问（如"nv diff 的"）大概率是承接上一轮话题的纠正，优先在上一轮候选基础上锁定具体目标
- 用户在读完长文后追问术语时，要把术语放回该文语境里解释（带上文中变体/对应关系），不要只给教科书定义
- cron schedule 调整后必须给出 cron id + 新表达式 + 含义的确认表，且确认"周末是否跑"这类边界
- 具身智能新闻归档每篇论文必须额外加 `- Project: <url>` 行（项目主页 > GitHub repo > demo，找不到写 N/A），供 awesome-physical-ai 自动 ingest，绝不可省略
- daily-papers 一句话总结模板：推荐文件路径（含篇数 + 必读/值得看/可跳过分层）+ 重点笔记篇数（含论文名 + 行数 + 质检通过）+ MOC 刷新状态（concept/paper 篇数，即使 config 关了 paper-reader Agent 也会顺带刷）
- arXiv 429 限流导致池子小（如仅 8 篇）属正常，无需特别归因或报错，按实际数报即可
